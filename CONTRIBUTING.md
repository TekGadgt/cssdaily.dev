# Contributing to cssdaily.dev

Thanks for your interest! cssdaily.dev is a small project I (@TekGadgt) maintain in my spare time. This document covers what kinds of contributions are welcome, how PRs get handled, and how contributors are recognized.

If you haven't already, skim the [README](./README.md) first — it covers the architecture, tech stack, and local dev setup, and the rest of this doc assumes you've seen it.

## The short version

- **Bug fixes and code improvements:** welcome, open a PR.
- **New features or larger changes:** open an issue first so we can talk about it.
- **New daily challenges:** not accepted — see below.
- **Everything else:** open an issue and we'll figure it out.

## About the challenges

Daily challenges for both modes are generated by Claude and curated by me — see [How It Works](./README.md#how-it-works) and [Daily Generation Pipeline](./README.md#daily-generation-pipeline) in the README for details. **I don't accept community-submitted challenges.** This keeps the difficulty curve, visual style, and scoring behavior consistent across the rotation.

What *is* welcome around challenges:

- Bug reports when a challenge behaves unexpectedly (wrong scoring, broken rendering, unsolvable, etc.)
- Feedback on difficulty or clarity
- Suggestions for new *categories* or *mechanics*, discussed in an issue
- Improvements to the generation prompts or pipeline (`scripts/generate-challenge.ts`, `scripts/generate-tailwind-challenge.ts`)

## What kinds of contributions are welcome

- **Bug fixes** — always welcome
- **Documentation improvements** — always welcome
- **Visual diff engine improvements** — welcome; these are subtle, please open an issue first
- **Editor improvements (CodeMirror 6 — transaction filters, decorations, autocomplete)** — welcome
- **Performance, a11y, and small UX polish** — welcome
- **GitHub Actions / daily cron improvements** — welcome
- **New features** — open an issue first. I'll decline features that expand the scope of the site beyond "daily CSS challenge" or that make the game feel less focused.
- **Refactors** — discuss first; I'm unlikely to accept large refactors without a clear motivation

Things I'll generally decline without discussion:

- PRs adding challenges
- Large reskins or redesigns
- New frameworks, state managers, or build tools
- Tests for code that isn't already tested (happy to discuss in an issue)

## Before you open a PR

**Open an issue first for anything non-trivial.** Typo fixes, obvious bugs, and small doc improvements are fine to send as a PR directly. For anything else, an issue saves us both time.

When filing a bug, please include:

- What you did
- What you expected
- What actually happened
- Browser + OS
- A link to the specific challenge (if relevant — e.g. `/challenge/2026-04-20` or `/tailwind/2026-04-20`)

## How PRs are handled

- I'll try to respond within a week, but life happens. If you haven't heard back in two weeks, bump the PR with a comment.
- I may request changes, suggest a different approach, or close PRs that don't fit the project's direction. If I close yours, I'll explain why — it's not personal.
- All changes to `main` go through a PR, including my own. The `main` branch is protected.
- Please keep PRs focused. One logical change per PR makes review easier and gets you merged faster.

## Development

Local setup is documented in the [README](./README.md#local-development). A few notes specific to contributing:

- **Generating challenges locally requires `ANTHROPIC_API_KEY`.** You don't need this for most work — the repo already contains committed challenge JSONs and target PNGs. You only need the key if you're actively working on the generation scripts.
- **The visual diff engine runs client-side.** Both the user's output and the target are rendered in hidden iframes and captured with snapdom, then compared pixel-by-pixel. Playwright is only used on the server to generate the initial target screenshot.
- **Tailwind mode uses sandboxed iframes** with class value sanitization. If you're touching that area, be mindful of XSS defense-in-depth.

## Recognition — all-contributors

This project uses [All Contributors](https://allcontributors.org/) to recognize everyone who helps, not just people who write code. Bug reports, feedback, docs, and ideas all count.

### Adding yourself after a contribution

Once your contribution has been accepted (PR merged, bug confirmed, etc.), comment on the relevant issue or PR with:

```
@all-contributors please add @YourGitHubUsername for <contribution-type>
```

For example:

```
@all-contributors please add @janedoe for code, doc
```

The bot will open a PR updating the README to include you. Combine multiple types with commas.

### Common contribution types

| Type | Emoji | What it means |
| --- | --- | --- |
| `code` | 💻 | Code contributions |
| `doc` | 📖 | Documentation |
| `bug` | 🐛 | Bug reports |
| `ideas` | 🤔 | Ideas, feedback, planning |
| `design` | 🎨 | Design work |
| `review` | 👀 | Reviewing PRs |
| `test` | ⚠️ | Tests |
| `a11y` | ♿️ | Accessibility |
| `infra` | 🚇 | Infrastructure (CI, build tooling, etc.) |
| `maintenance` | 🚧 | Repo maintenance |

The full list is in the [Emoji Key](https://allcontributors.org/docs/en/emoji-key). If you're not sure what type to use, just ask in your comment.

If you'd rather not be added, let me know.

## Code of conduct

Be kind. Assume good faith. If you wouldn't say it to someone's face, don't say it in an issue or PR.

## Questions

If anything here is unclear, open an issue and ask. This doc will evolve as the project does.
